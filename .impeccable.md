This file is the canonical design-language document for `gtm-command-center`.

It defines the product's design intent: brand personality, visual language, typography, motion, color usage, composition heuristics, and anti-patterns to avoid.

Implementation details like tokens, component classes, and shared UI conventions belong in `CLAUDE.md` and `globals.css`.

## Design Context

### Users

Single user: Omar Nasser, a GTM Engineer running his job search pipeline. He uses this app daily to analyze job descriptions, draft outreach emails, research companies/people, run coaching sessions, and maintain a decision trail. Context is always focused, task-oriented work — paste data in, get structured intelligence out. Speed and clarity matter more than discovery or exploration.

### Brand Personality

**Calm, precise, confident.** Like a well-organized analyst's desk — quiet authority, no flash. The interface should feel like a trusted instrument, not a product trying to impress. Every element earns its place.

### Aesthetic Direction

- **Primary reference:** Linear — clean data density, keyboard-first, subtle animations, monochrome + accent
- **Visual tone:** Restrained, warm-neutral, typographically precise
- **Anti-references:** Overly playful SaaS dashboards, gradient-heavy marketing tools, anything with rounded cartoon illustrations or excessive color
- **Theme:** Light + dark mode (warm neutrals in light, true dark in dark mode)
- **Palette:** Warm beige/off-white base (`#f7f7f5`), blue accent (`#1e63ff`), semantic greens/oranges/reds for status. Dark mode: near-black backgrounds, same accent and semantic colors adjusted for dark surfaces
- **Typography:** System font stack (SF Pro Text), tight hierarchy — `text-xl` page titles, `text-sm` body, `text-xs` metadata
- **Icons:** Lucide React, 14-16px standard sizes, muted color unless interactive

### Design Principles

1. **Data density over decoration** — Show more information in less space. No decorative elements, no unnecessary whitespace padding. Every pixel serves comprehension.

2. **Status at a glance** — Verdicts, scores, and states should be immediately scannable via color-coded badges, position, and typographic weight. A user glancing at a list should know what needs attention without reading.

3. **Keyboard-first, mouse-friendly** — Command palette (Cmd+K) is the primary navigation. All interactive elements must have visible focus states. Animations should be subtle (120ms transitions, not 300ms bounces).

4. **Consistency is kindness** — Same spacing, same border-radius, same badge patterns everywhere. When the user learns one pattern, it should transfer to every other view. Extract and reuse, never duplicate.

5. **Quiet confidence** — The UI should feel like it knows what it's doing. No loading spinners without context. No empty states without guidance. No errors without actionable next steps. Calm, never anxious.

### Typography Rules

- Use the system font stack only. Favor clarity and rhythm over expressiveness.
- Keep hierarchy tight and predictable: page titles at `text-xl`, section titles around `text-sm` to `text-base`, body copy at `text-sm`, and metadata at `text-xs`.
- Use weight and color before size inflation. A glanceable dashboard should feel dense, not poster-like.
- Reserve large text for truly important moments: page headers, score/value emphasis, and primary empty-state messages.
- Prefer one clean reading width and short supporting copy. Multi-line text should earn its space.
- Avoid decorative typography moves: serif display fonts, gradient text, all-caps overload, or oversized numbers that act as ornament rather than information.

### Motion Rules

- Motion exists to explain state change, confirm interaction, or guide attention. If it does none of those, remove it.
- Default to subtle transitions in the `120ms-200ms` range. Prefer calm ease-out timing over bounce or theatrical reveals.
- Interaction-driven motion is good: hover, focus, expand/collapse, loading-state transitions, optimistic updates.
- Passive entrance animation should be rare. Static dashboards should not feel like a landing page.
- Respect reduced motion every time. No animation should be required to understand the interface.
- Avoid animating layout for spectacle. Motion should support comprehension, not make dense UI feel slippery.

### Color Usage Principles

- The palette is mostly warm neutrals with blue as the primary interaction accent and semantic green/orange/red for status.
- Color should answer one of three questions: "what can I act on?", "what state is this in?", or "what deserves attention first?"
- Reserve blue for interaction, focus, links, and selected states. Do not spend it casually on decorative surfaces.
- Reserve semantic colors for meaning, not decoration. A warning color should indicate warning, not just visual variety.
- Most surfaces should stay neutral. Let color punctuate the interface rather than flood it.
- Avoid same-hue clustering, repeated colorful pills, and colored borders on already colored surfaces. If everything is accented, nothing is.

### Composition and List Heuristics

- Prefer flat, scannable layouts over nested boxes. Use spacing, alignment, and text hierarchy before adding containers.
- Cards are for truly distinct, actionable units. Do not wrap every row in extra chrome just because it is convenient.
- Never nest cards inside cards unless there is a strong interaction boundary.
- Lists should create hierarchy at a glance: primary label first, secondary metadata demoted, key status or counts easy to scan.
- Repeated rows must vary by information, not just by copy. If the same badge or timestamp appears on nearly every item, it is probably noise.
- Use horizontal space deliberately. Dense product UI should not leave the useful content crammed into one corner while the rest of the row sits empty.
- Repetition is acceptable when it teaches a pattern. Repetition is bad when it flattens importance.

### Anti-Patterns

- Gradient-heavy UI, decorative glass, or glow effects that make the app feel like a marketing site
- Decorative icon boxes, especially when the icon adds no meaning
- Repeated zero-information badges, such as a label shown on every row
- Nested cards, stacked wrappers, or too many bordered containers creating flat hierarchy
- Identical rows with the same metadata treatment, causing a wall-of-sameness effect
- Oversized numbers, badges, or accent colors that dominate without carrying the most important information
- Redundant headings or explanatory copy that repeat what the user can already see
- Loading states without context, empty states without guidance, or error states without a next step

### Technical Constraints

- Supabase backend
- Vercel deployment
- System fonts only (no custom font loading)
- Light + dark mode via CSS variables
